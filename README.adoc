= CloudBees action: Scan with Snyk SCA

Use this action to scan application dependencies for security vulnerabilities with the Snyk SCA (Software Composition Analysis) scanner.
You can also use the action output as a quality gate for the next step or job in your workflow.

Snyk Software Composition Analysis (SCA) is a developer-first security solution that scans and monitors open source dependencies for security vulnerabilities and license issues.

Add a Snyk SCA scan to your workflows in CloudBees platform to:

* Detect security vulnerabilities in open source dependencies.
* Identify outdated packages and dependencies with known vulnerabilities.
* Gain insight into dependency security risks and how to fix issues.
* Ensure compliance with security best practices and licensing requirements.
* Shift security left by catching issues early in the development lifecycle.

CloudBees platform enables you to run a Snyk SCA scan either implicitly or explicitly.

== Explicit and implicit scan types

An implicit scan is automatically triggered, and an explicit scan is one you configure to be invoked in a step of your workflow.
To learn more about the differences between explicit and implicit scans, refer to link:https://docs.cloudbees.com/docs/cloudbees-platform/latest/actions#security[Security scan actions].

To set up implicit scanning, refer to link:https://docs.cloudbees.com/docs/cloudbees-platform/latest/aspm/implicit-security-analysis[Code and binary security analysis].

== Scanner information

The Snyk SCA scanner architectural components are:

* Snyk CLI: The command-line interface that performs the scanning.
* Snyk Vulnerability Database: Comprehensive database of known vulnerabilities in open source packages.
* Snyk Policy Engine: Evaluates dependencies against security rules and license policies.

The scanning process is as follows:

. The Snyk SCA scanner identifies dependency manifest files in your repository (package.json, pom.xml, requirements.txt, etc.).
. Dependencies are parsed and analyzed against Snyk's vulnerability database.
. Security vulnerabilities and license issues are identified based on known CVEs and security advisories.
. Results are reported with severity levels, detailed descriptions, and remediation guidance.
. Results are made available as actionable outputs for quality gates and downstream workflow steps.

NOTE: For more information about Snyk SCA, refer to the link:https://docs.snyk.io/products/snyk-open-source[Snyk Open Source documentation].

== Inputs

[cols="2a,1a,1a,3a",options="header"]
.Input details
|===

| Input name
| Data type
| Required?
| Description

| `snyk-token`
| String
| Yes
| The Snyk API token for authentication.

| `org-id`
| String
| Yes
| The Snyk organization ID.

| `cli-params`
| String
| No
| Additional CLI parameters to pass to the Snyk CLI.

| `build-directory`
| String

| String
| No

== Usage examples

=== Basic usage

The following is a basic usage example for this action:

[source,yaml]
----
      - name: Scan with Snyk SCA
        uses: cloudbees-io/snyk-sca-plugin-private@v1
        with:
          snyk-token: ${{ secrets.SNYK_TOKEN }}
          org-id: ${{ vars.SNYK_ORG_ID }}
          ref: main

----

=== Scan a Node.js project with Snyk SCA

The following CloudBees platform workflow example scans a Node.js project with Snyk SCA.

[source, yaml,role="default-expanded"]
----
name: snyk-sca-scan
kind: workflow
apiVersion: automation.cloudbees.io/v1alpha1

on:
  push:
    branches:
      - main

permissions:
  scm-token-own: read
  scm-token-org: read
  id-token: write

jobs:
  snyk-sca-scan:
    steps:
      - name: Check out Node.js source code
        uses: cloudbees-io/checkout@v1

      - name: Snyk SCA scan on Node.js dependencies
        uses: cloudbees-io/snyk-sca-plugin-private@v1
        with:
          snyk-token: ${{ secrets.SNYK_TOKEN }}
          org-id: ${{ vars.SNYK_ORG_ID }}
          ref: main

----

=== Using the action output

Access the output values in downstream steps and jobs using the `outputs` link:https://docs.cloudbees.com/docs/cloudbees-platform/latest/dsl-syntax/contexts[context].

Use the output in your workflow as follows, where <action_step_ID> is the action step ID, and <severity> is an output parameter name, such as `critical-count`:

[source,yaml]
----
${{steps.<action_step_ID>.outputs.<severity>}}
----

The following example uses the action output in a downstream step of the same job:

[source,yaml,role="default-expanded"]
----
name: snyk-sca-with-downstream-step
kind: workflow
apiVersion: automation.cloudbees.io/v1alpha1

on:
  push:
    branches:
      - main

permissions:
  scm-token-own: read
  scm-token-org: read
  id-token: write

jobs:
  snyk-sca-scan-job:
    steps:
      - name: Check Out Source Code
        uses: cloudbees-io/checkout@v1

      - id: snyk-sca-step
        name: Snyk SCA Scan
        uses: cloudbees-io/snyk-sca-plugin-private@v1
        with:
          snyk-token: ${{ secrets.SNYK_TOKEN }}
          org-id: ${{ vars.SNYK_ORG_ID }}
          ref: main

      - name: Examine Source Directory
        uses: docker://golang:1.20.3-alpine3.17
        shell: sh
        run: |
          ls -latR /cloudbees/workspace

      - id: print-outputs-from-snyk-sca-step
        name: Print Outputs From Upstream Snyk SCA Step
        uses: docker://alpine:latest
        run: |
            # Printing all outputs
            echo "Outputs from upstream Snyk SCA step:"
            echo "Critical count: ${{steps.snyk-sca-step.outputs.critical-count}}"
            echo "Very high count: ${{steps.snyk-sca-step.outputs.very-high-count}}"
            echo "High count: ${{steps.snyk-sca-step.outputs.high-count}}"
            echo "Medium count: ${{steps.snyk-sca-step.outputs.medium-count}}"
            echo "Low count: ${{steps.snyk-sca-step.outputs.low-count}}"


----

The following example uses the action output in a downstream job:

[source,yaml,role="default-expanded"]
----
name: snyk-sca-with-downstream-job
kind: workflow
apiVersion: automation.cloudbees.io/v1alpha1

on:
  push:
    branches:
      - main

permissions:
  scm-token-own: read
  scm-token-org: read
  id-token: write

jobs:
  job1:
    outputs:
      snyk-sca-job-output-critical: ${{ steps.snyk-sca-step.outputs.critical-count }}
      snyk-sca-job-output-very-high: ${{ steps.snyk-sca-step.outputs.very-high-count }}
      snyk-sca-job-output-high: ${{ steps.snyk-sca-step.outputs.high-count }}
      snyk-sca-job-output-medium: ${{ steps.snyk-sca-step.outputs.medium-count }}
      snyk-sca-job-output-low: ${{ steps.snyk-sca-step.outputs.low-count }}
    steps:
      - name: Check Out Source Code
        uses: cloudbees-io/checkout@v1
        with:
          repository: my-gh-repo-org/my-repo
          ref: main
          token: ${{ secrets.GIT_PAT }}

      - id: snyk-sca-step
        name: Snyk SCA Scan
        uses: cloudbees-io/snyk-sca-plugin-private@v1
        with:
          snyk-token: ${{ secrets.SNYK_TOKEN }}
          org-id: ${{ vars.SNYK_ORG_ID }}
          ref: main

  job2:
    needs: job1
    steps:
      - id: print-outputs-from-job1
        name: Print Outputs From Upstream Job1
        uses: docker://alpine:latest
        run: |
          # Printing all outputs
          echo "Outputs from upstream Snyk SCA job:"
          echo "Critical count: ${{ needs.job1.outputs.snyk-sca-job-output-critical }}"
          echo "Very high count: ${{ needs.job1.outputs.snyk-sca-job-output-very-high }}"
          echo "High count: ${{ needs.job1.outputs.snyk-sca-job-output-high }}"
          echo "Medium count: ${{ needs.job1.outputs.snyk-sca-job-output-medium }}"
          echo "Low count: ${{ needs.job1.outputs.snyk-sca-job-output-low }}"

----
